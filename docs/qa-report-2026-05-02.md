## Findings — fixed during this audit

### 1. ✅ FIXED — Horizontal overflow on `/portal/lab/blueprint` at every viewport (was CRITICAL)

**Symptom:** the Blueprint canvas scrolled horizontally on every viewport, including desktop. Chapter prose was being cut off by the right edge of the viewport. Visible to every Tier 1 customer the moment they opened the new flow.

**Root cause (three layers):**

1. The Opus 4.7 chapter-draft pipeline emits a trailing fenced ```` ```json ```` block when it cooperates with the prompt's provenance contract. React-Markdown renders this as `<pre><code>` which doesn't wrap by default. The `<code>` block's intrinsic width (~1900px for a multi-line JSON manifest) blew out the page width.
2. The chapter-card grid track was `1fr` without `min-w-0`, so it expanded to its content's intrinsic width instead of shrinking to its share of the row.
3. No defensive `overflow-x: auto` on `<pre>` elements inside `.chapter-prose`, so any future code blocks would have hit the same bug.

**Fixes shipped:**

- `src/components/agent/ChapterCard.tsx`: strip the trailing ```` ```json ```` block from the render (it's already in the provenance panel — duplicate noise to the customer). Also added `overflow-wrap: anywhere` and `word-break: break-word` for long URLs/tokens (the brand_voice scrape leaves OG image URLs in content), and `pre { max-width: 100%; overflow-x: auto; }` for any future code blocks.
- `src/app/portal/lab/blueprint/page.tsx`: added `min-w-0` to the chapter-cards grid track with a comment explaining the constraint.

**Verified:** at 375px viewport, all 16 chapter articles now uniformly clamp to the available width (325px content), `htmlScrollWidth` matches `innerWidth` (no overflow), prose wraps cleanly. Desktop unchanged.

### 2. ✅ FIXED — Exclamation in newsletter welcome plain-text fallback

`src/lib/email/templates/newsletter-welcome.tsx:104` — `Welcome to The Franchisor Blueprint blog!` → period. The React render side already used "Welcome to the list" (no `!`), only the plain-text fallback was wrong.

### 3. ✅ FIXED — `Capability 04` legacy label in marketing mockup

`src/components/DeviceMockups.tsx:342` — `Capability 04` → `Mastery 04`. Single instance missed during the previous Capability→Mastery rename.

### 4. ✅ FIXED — Three Phase 1 animations didn't respect `prefers-reduced-motion`

WCAG 2.3.3 violation: customers with vestibular sensitivity would see the `TypedHeading` caret blinking, the `JasonChatDock` dock breathing, and the `IntakeClient` CTA pulsing regardless of OS-level motion preference.

Fixed in three files:
- `src/components/agent/TypedHeading.tsx` — both the JS-side typing animation AND the CSS-side caret blink now disable when `(prefers-reduced-motion: reduce)` matches. The full text renders immediately and the caret hides.
- `src/components/agent/JasonChatDock.tsx` — `.jason-dock-breathe` CSS rule now no-ops under reduced-motion.
- `src/app/portal/lab/intake/IntakeClient.tsx` — `.active-pulse` CSS rule now no-ops under reduced-motion.

The pre-existing `tfb-stage-in` animation on the assessment flow already had this rule — the new ones now match.

### 5. ✅ FIXED — Provenance panel radius inconsistency

`src/components/agent/ChapterCard.tsx:215` — `rounded-lg` (8px) inside a `rounded-2xl` (16px) parent card. Bumped to `rounded-xl` (12px) — visually nested-feeling without competing with the parent.

## Findings — surfaced for product decision (not auto-fixed)

These are debate items, not bugs. Eric/Jason should make a call, then I (or a future Claude) can apply.

### 6. MEDIUM — Comparison emoji ✅/❌ in long-form blog MDX

`src/app/blog/(posts)/the-real-cost-of-franchising-your-business/page.mdx` lines 128–171 use checkmark and X emoji as bullet markers in a "DIY vs Outsourced vs Done-For-You" comparison.

The TFB voice rule says "no emoji decoration in headings, CTAs, microcopy, or body." Comparison lists in long-form content are arguably the closest emoji come to being legitimate (they're functional, not decorative). But strict compliance would replace them.

**Three options:**

1. **Replace with Lucide icons** (`<Check size={14} className="text-emerald-500" />` / `<X size={14} className="text-red-700" />`) — aligns with the rest of the icon vocabulary, but requires custom MDX components.
2. **Replace with plain text** ("Pro:" / "Con:" prefixes) — simplest, most strict-compliant.
3. **Carve out an exception** for blog comparison lists — keep as-is, document in `docs/jason-agent-prompt.md`.

I'd lean (1) — it's the most TFB-correct, and it's a 30-minute MDX component setup we'd reuse on future blog posts. Awaiting your call.

### 7. MEDIUM — `font-light` (300) on hero subheadings across 11 marketing pages

The `tfb-designers-eye` skill says "no weights below 400 in body copy" — but `font-light` is being used intentionally on cream-on-navy hero subtitles where the lighter weight gives an editorial feel. Linear and similar premium products do this too at large sizes (text-lg+).

**Files affected:** `src/app/page.tsx` (3 instances), `src/app/about/page.tsx` (2), `src/app/assessment/page.tsx`, `src/app/blog/(posts)/layout.tsx`, `src/app/programs/page.tsx`, `src/app/programs/blueprint/page.tsx`, `src/app/pricing/page.tsx`.

**Two options:**

1. **Strict compliance**: change all 11 to `font-normal`. Marginal weight bump, slightly heavier feel, fully on-spec.
2. **Carve out an exception**: hero subtitles (hero context, text-lg+ sizes, on dark backgrounds) can use `font-light`. Document in the skill spec and `docs/jason-agent-prompt.md`.

I'd lean (2) — the current treatment IS premium-coded and changing it loses something. But Jason's call.

### 8. MEDIUM — Mobile users have no way to navigate between chapters on `/portal/lab/blueprint`

The sidebar TOC is `hidden md:block`. On mobile, a user has to scroll through 16 chapters linearly to find the one they want. The Blueprint canvas was designed mobile-aware enough that the chapters all render correctly, but discovery is broken.

**Suggested fix (Phase 1.5 polish, ~1 hour):**

A bottom-of-screen "Jump to chapter" pill that, when tapped, opens a sheet/drawer listing all 16 chapters with their fill state — same data as the desktop sidebar TOC, just transient on mobile. Or a sticky `<select>` that scroll-targets the matching `#chapter-X` anchor.

### 9. MEDIUM — In-flight draft (`/api/agent/draft`) survives a page reload silently

If a customer clicks "Draft with Jason" and refreshes the page mid-draft (60-90s), the page reloads with no indication that a draft is still being generated server-side. The draft completes and writes to Memory, but the customer never sees the "your chapter just appeared" moment.

**Suggested fix:** track in-flight drafts in `customer_memory_drafts` (new table) with a `started_at` timestamp; on Blueprint canvas load, show a "Drafting in progress (kicked off N seconds ago)" pill on any chapter mid-draft. Or stream the draft response so the customer sees it materialize live (the bigger fix, Phase 1.5).
